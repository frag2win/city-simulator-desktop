## 1. Executive Summary

The **Procedural 3D City Simulator** is a cross-platform desktop application that transforms raw OpenStreetMap geographic data into navigable, interactive 3D urban environments — offline-capable, installable, and fully native to Windows, macOS, and Linux.

Built on **Electron** as its desktop shell, the application embeds a **Python geospatial processing engine** (accessed via a local child process / REST sidecar) alongside a **Three.js 3D renderer** running inside the Electron renderer process. The entire stack is packaged into a single distributable installer using `electron-builder`.

The desktop-first architecture unlocks three critical capabilities unavailable to web applications:

- **Unrestricted local compute** — no browser memory caps, full CPU/GPU access for city-scale rendering and simulation
- **Offline operation** — once a city is ingested and cached to the local SQLite/SpatiaLite database, the user requires no internet connection
- **Native OS integration** — file system access for exporting simulation data, native menus, OS notifications, and deep-link support

This document defines the complete product requirements: problem, solution, tech stack, file structure, phased implementation roadmap, IPC contracts, and acceptance criteria.

## 2. Problem Statement

### 2.1 The Static Map Limitation

Modern mapping technology, despite immense investment, remains fundamentally **passive and non-reactive**. Geographic data is abundant but siloed, and current tools fail to surface the relationships between physical infrastructure, human movement, and systemic events.

#### Three Core Failure Modes

| Failure Mode | Description |
|---|---|
| **Data Fragmentation** | Building heights, road capacities, utility networks, and land-use classifications exist in massive, disjointed datasets (OpenStreetMap, government GIS portals, satellite imagery) with no unified layer for visualization or simulation. |
| **The Modeling Bottleneck** | Creating a navigable 3D city model manually requires hundreds of hours of 3D-artist labor per square kilometer. This is technically and economically unfeasible for most organizations and individual researchers. |
| **Lack of Consequence** | Static maps cannot model causality. They cannot answer: *How does a traffic jam on Avenue X degrade air quality in Block Y? How does a power failure at Substation Z cascade through an urban district?* |

### 2.2 The Desktop Gap

Existing geospatial tools (QGIS, ArcGIS, Google Earth Pro) are either limited to 2D analysis, require expensive proprietary licenses, demand constant cloud connectivity, or fail to combine real-time simulation with interactive 3D visualization in a single offline-capable package.

A purpose-built desktop application closes this gap by combining data ingest, spatial processing, 3D rendering, and agent simulation into one installable tool that runs entirely on the user's machine.

## 3. Solution Overview

### 3.1 Core Concept: The Procedural Digital Twin

The simulator assembles a **living 3D model** of any geographic area using four tightly integrated subsystems. "Procedural" means the 3D world is generated by code logic and mathematical rules applied to real-world data — not drawn by hand.

| Subsystem | Technology | Function | Reasoning |
|---|---|---|---|
| **Data Ingestion** | Python + Overpass API | Query and normalize OSM geographic data; persist to local DB | Overpass API is the most granular freely accessible urban dataset. Local persistence enables full offline operation after first load. |
| **Spatial Projection** | PyProj + Custom Math | Convert lat/lon to local 3D Cartesian coordinates | Web Mercator projection prevents visual distortion at city scale. Done in Python to leverage mature geospatial math libraries. |
| **Procedural Extrusion** | Three.js ExtrudeGeometry | Generate 3D meshes from 2D polygons using OSM metadata | Eliminates 3D-artist dependency. Any building with a `building:levels` tag becomes a precise mesh automatically. |
| **Agent Simulation** | A\* Pathfinding + Python | Simulate cars, pedestrians, and infrastructure events | Turns the city from a static model into a reactive environment capable of answering causal "what-if" questions. |

### 3.2 Technical Approach

#### 3.2.1 Automated Data Ingestion
The Python sidecar process queries the **Overpass API** using a bounding-box strategy, fetching OSM `ways` (closed polygons: buildings, parks, water) and `nodes` (point data: intersections, amenities). Each feature's metadata tags — `building:levels`, `height`, `highway`, `landuse`, `amenity` — are extracted, validated, and written into a local **SpatiaLite** database. Subsequent loads of the same city are served entirely offline from this cache.

#### 3.2.2 Spatial Projection Logic
Latitude/longitude coordinates are converted into a flat local X/Y/Z Cartesian grid using a **Web Mercator projection** anchored to the centroid of the queried bounding box. All coordinates are expressed as meter-offset vectors from this origin, ensuring geometric fidelity across city-scale distances without distortion.

#### 3.2.3 Procedural Extrusion
The Electron renderer process consumes building footprint polygons via IPC and extrudes them along the Z-axis using `THREE.ExtrudeGeometry`. Extrusion height = `building:levels × 3.5 meters`. Buildings without a levels tag receive a probabilistic default height based on surrounding context. The entire city mesh is generated in under 3 seconds for a 1 km² area.

#### 3.2.4 Agent-Based Simulation
OSM `highway` edges are extracted into a weighted graph. The **A\* algorithm** generates optimal paths between origin-destination pairs. Virtual agents (cars, pedestrians) traverse these paths in real time, rendered as animated 3D objects. Dynamic events — road closures, power outages, density surges — alter graph weights and trigger visual state changes across connected entities, all computed inside the Python sidecar and streamed to the renderer via IPC.

## 4. Why a Desktop Application

This is a deliberate architectural decision, not a constraint. The following table maps the desktop-first rationale directly against the technical requirements of the project.

| Requirement | Web App Limitation | Desktop Advantage |
|---|---|---|
| **City-scale 3D rendering** | Browser memory limits (~2GB), WebGL context loss on large scenes | Full system RAM/GPU access; no context loss; can render 10+ km² |
| **Offline operation** | Requires constant CDN and API connectivity | City data cached locally in SpatiaLite; fully air-gapped after first sync |
| **Python processing engine** | Cannot run a Python process in a browser | Electron spawns Python as a native child process with full OS access |
| **Heavy geospatial computation** | Web Workers are sandboxed and limited | Python sidecar runs on all CPU cores; NumPy/Shapely operate at native speed |
| **File system I/O** | Sandboxed File System Access API; no arbitrary path access | Full `fs` module access for reading/writing simulation exports, GeoJSON, screenshots |
| **Native UI integration** | No access to OS menus, tray, notifications, or file associations | Native menus, dock icons, system tray, OS notifications, `.city` file associations |
| **Distribution model** | No installation; no persistent local state | Single installer (`.exe` / `.dmg` / `.AppImage`); app persists state between sessions |

## 5. Technology Stack

### 5.1 Core Stack

| Layer | Technology | Version | Justification |
|---|---|---|---|
| **Desktop Shell** | Electron | 30.x | Industry-standard Chromium + Node.js desktop framework. Provides native OS integration, IPC bridge, and cross-platform packaging. |
| **UI Framework** | React.js | 18.x | Component-based UI running in the Electron renderer process. Fast state reconciliation ideal for real-time data-driven panel updates. |
| **3D Rendering** | Three.js | r165+ | WebGL abstraction with `ExtrudeGeometry`, instanced meshes, and shader support. Runs in Electron's renderer with full GPU access — no browser memory cap. |
| **State Management** | Zustand | 4.x | Lightweight store for managing simulation state, selected entities, scene config, and UI panel visibility. |
| **Desktop Build** | electron-builder | 24.x | Packages the app into `.exe` (NSIS installer), `.dmg` (macOS), and `.AppImage` / `.deb` (Linux) from a single config. |
| **IPC Bridge** | Electron IPC (contextBridge) | Built-in | Secure communication channel between renderer process (Three.js/React) and main process (Python sidecar, file system). |
| **Python Runtime** | Python (embedded via pyenv / bundled) | 3.11+ | Sidecar process for geospatial computation. Bundled with the app using `pyinstaller` to avoid requiring user to install Python separately. |
| **Python Framework** | FastAPI (local sidecar) | 0.111+ | Python sidecar exposes a localhost REST + WebSocket server on a random port. Electron main process spawns it as a child process on app launch. |
| **Geospatial Processing** | Shapely + PyProj | 2.x / 3.x | Shapely handles polygon operations (centroid, buffer, simplification, validation); PyProj manages coordinate reference system transforms. |
| **Local Database** | SpatiaLite (SQLite extension) | 5.x | Zero-config embedded spatial database. Stores ingested OSM data locally. No separate DB server required — fits the desktop deployment model perfectly. |
| **Caching** | SQLite (via SpatiaLite) | Built-in | City bounding-box cache with TTL metadata. Offline-first: app checks local DB before attempting Overpass API query. |
| **Async Tasks** | Python asyncio + threading | Built-in | Large city ingestion jobs run in background threads inside the Python sidecar, streaming progress events to Electron via WebSocket. |
| **Testing — Frontend** | Vitest + Playwright | Latest | Vitest for React/Three.js unit tests; Playwright with `electron-playwright-helpers` for E2E desktop automation tests. |
| **Testing — Backend** | Pytest | 8.x | Unit and integration tests for the Python sidecar: Overpass client, spatial processor, simulation engine. |
| **CI/CD** | GitHub Actions | Latest | Cross-platform matrix builds: Windows (windows-latest), macOS (macos-latest), Linux (ubuntu-latest). Auto-publishes installers on tagged releases. |

### 5.2 Key Electron Architecture Decisions

| Decision | Choice | Reason |
|---|---|---|
| **Process model** | Multi-process (main + renderer + Python sidecar) | Isolates heavy computation from UI thread; crash in Python sidecar does not freeze the renderer |
| **IPC security** | `contextBridge` + `preload.js` (no `nodeIntegration`) | Follows Electron security best practices; renderer cannot access Node.js APIs directly |
| **Python packaging** | PyInstaller bundled binary | Users do not need Python installed; sidecar binary is included in the installer |
| **Database location** | `app.getPath('userData')` | OS-appropriate user data directory (e.g., `%APPDATA%` on Windows, `~/Library/Application Support` on macOS) |
| **Auto-update** | `electron-updater` | Delta updates pushed via GitHub Releases; users notified in-app |

## 11. IPC & Internal API Contract

The desktop application uses two communication channels between the Electron renderer and the Python sidecar.

### 10.1 Electron IPC Channels (Renderer ↔ Main Process)

| Channel | Direction | Payload | Description |
|---|---|---|---|
| `city:load` | Renderer → Main | `{ bbox: "N,S,E,W" }` | Request city data load. Main checks SpatiaLite cache, then queries sidecar. |
| `city:loaded` | Main → Renderer | `{ geojson: FeatureCollection }` | City GeoJSON ready for scene construction. |
| `city:progress` | Main → Renderer | `{ stage: string, percent: number }` | Forwarded from Python WebSocket; drives ProgressModal. |
| `simulation:start` | Renderer → Main | `{ agentCount, speed }` | Instructs Python sidecar to begin simulation. |
| `simulation:event` | Renderer → Main | `{ type: "road_closure" \| "power_outage", entityId }` | User-triggered event forwarded to sidecar. |
| `simulation:state` | Main → Renderer | `{ agents[], stats, time }` | Forwarded from Python WebSocket at 10fps. |
| `file:export` | Renderer → Main | `{ format: "geojson" \| "city" }` | Opens native save dialog; writes file to selected path. |
| `file:open` | Main → Renderer | `{ path: string }` | Triggered when user opens a `.city` file association. |
| `app:updateAvailable` | Main → Renderer | `{ version: string }` | Triggers `UpdateNotice.jsx` banner in renderer. |
| `city:error` | Main → Renderer | `{ code: string, message: string, recoverable: boolean }` | Signals a city load or ingestion failure. Renderer shows error toast. |
| `simulation:error` | Main → Renderer | `{ code: string, message: string }` | Signals a simulation fault (e.g., sidecar crash, graph build failure). |
| `sidecar:status` | Main → Renderer | `{ status: "starting" \| "ready" \| "crashed" \| "restarting" }` | Sidecar lifecycle updates; drives status indicator in UI. |

### 10.2 Python Sidecar Localhost API

The sidecar runs on a random port communicated to Electron at startup via stdout.

| Method | Endpoint | Parameters | Response | Description |
|---|---|---|---|---|
| GET | `/health` | — | `{ status, db, version }` | Sidecar readiness check on startup. |
| GET | `/city` | `bbox=N,S,E,W` | GeoJSON FeatureCollection | Fetch normalized city features. Served from SpatiaLite if cached. |
| GET | `/city/cache` | — | `[{ name, bbox, size_mb, cached_at }]` | List all cached cities for the cache manager UI. |
| DELETE | `/city/cache/{id}` | — | `{ deleted: true }` | Delete a specific cached city record and data. |
| WebSocket | `/ws/simulate` | `bbox, agent_count, speed` | JSON frames: `{ agents, events, stats, time }` | Real-time simulation state stream at 10fps. |
| WebSocket | `/ws/ingest` | `bbox` | JSON frames: `{ stage, percent }` | Progress events during city ingestion. |
| POST | `/auth/token` | `{ token: string }` | `{ valid: bool }` | Validates the one-time auth token passed by Electron at sidecar startup. All other endpoints require `Authorization: Bearer <token>` header. |

## 12. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation Strategy |
|---|---|---|---|
| **Overpass API rate limiting or downtime** | Medium | High | SpatiaLite cache serves data offline after first load. App displays clear "Offline Mode" status. Allow manual GeoJSON file import as fallback. |
| **OSM data quality variance between cities** | High | Medium | Robust Pydantic tag validation with intelligent defaults. Data-quality score surfaced in UI so users understand confidence level of rendered heights. |
| **WebGL performance degradation on large cities** | High | High | Three.js `InstancedMesh` for all buildings. Level-of-Detail (LOD) — simplify geometry > 500m from camera. Frustum culling for off-screen objects. Scene chunking for cities > 3km². |
| **Python sidecar crash / unresponsive** | Medium | High | Electron main process monitors sidecar via health check ping every 5 seconds. Auto-restart on crash with toast notification to user. Circuit breaker prevents restart loop. |
| **Electron app installer size bloat** | Medium | Medium | Use `electron-builder` tree-shaking. Bundle only required Python packages via PyInstaller `--onedir`. Compress assets. Target < 250MB per platform. |
| **A\* performance on dense urban road graphs** | Medium | Medium | Pre-compute and cache road graph in SpatiaLite at ingest time. Use bidirectional A\* for long paths. Cache recent path lookups in Redis-like in-memory dict in Python. |
| **Cross-platform UI inconsistencies** | Medium | Low | Automated Playwright tests run on all three platform targets in CI. Custom title bar avoids platform-specific native frame differences. |
| **macOS notarization failure blocking App Store / Gatekeeper** | Low | High | Set up Apple Developer notarization in CI pipeline early (Phase 1). Test `entitlements.mac.plist` before any release candidate. |

## 15. Appendix: Key Definitions

| Term | Definition |
|---|---|
| **Electron** | A framework for building cross-platform desktop apps using web technologies (Chromium + Node.js). Used as the application shell, providing native OS integration, window management, and IPC. |
| **IPC (Inter-Process Communication)** | The mechanism in Electron for secure message passing between the main process (Node.js, OS access) and the renderer process (React/Three.js). All communication goes through `contextBridge` in `preload.js`. |
| **Python Sidecar** | A separate Python process spawned by Electron on startup. It runs a local FastAPI server on a random localhost port, handling all geospatial computation and simulation logic. |
| **SpatiaLite** | An extension to SQLite that adds support for geographic (spatial) objects. Used as the embedded local database — zero-config, no server required, ideal for desktop deployments. |
| **PyInstaller** | A tool that bundles a Python application and all its dependencies into a single executable binary, so end users do not need Python installed on their system. |
| **electron-builder** | A build tool that packages an Electron app into platform-specific installers: NSIS `.exe` (Windows), `.dmg` (macOS), `.AppImage` / `.deb` (Linux). |
| **Overpass API** | A read-only API serving custom-selected parts of OpenStreetMap data. Used as the primary geographic data source, queried via the Python sidecar. |
| **OSM (OpenStreetMap)** | An open, collaborative map of the world providing raw building footprints, road networks, and geographic metadata that the system consumes. |
| **Web Mercator** | The de facto standard coordinate reference system for web mapping (EPSG:3857). Projects latitude/longitude onto a flat 2D plane for local Cartesian rendering. |
| **Procedural Generation** | The algorithmic creation of 3D content using code and mathematical rules rather than manual artistic creation. A building `levels` tag becomes a precise 3D mesh automatically. |
| **ExtrudeGeometry** | A Three.js class that takes a 2D shape (polygon) and extrudes it along the Z-axis to create a 3D solid — the core mechanism for building mesh generation. |
| **A\* (A-Star)** | A graph search algorithm that finds the shortest weighted path between two nodes. Used for agent navigation on the road network graph. |
| **Agent-Based Simulation** | A computational model where autonomous entities ("agents") interact with their environment according to defined rules — here, cars and pedestrians navigating a city. |
| **Digital Twin** | A virtual model that accurately reflects a physical object or system. In this context, a 3D data-aware representation of a real urban area built from live geographic data. |
| **contextBridge** | An Electron API that exposes a safe, limited API from the main process to the renderer process, without enabling full Node.js access in the renderer (security best practice). |
| **LOD (Level of Detail)** | A 3D rendering optimization where distant objects use simpler geometry than nearby objects, reducing GPU load without visible quality loss at the camera's current position. |

## 15.1 `.city` File Format Specification

The `.city` file is a **zipped JSON archive** (`.city` = renamed `.zip`) containing:

```
mycity.city  (zip archive)
├── manifest.json       # Format version, app version, creation timestamp
├── city.geojson        # Full normalized GeoJSON FeatureCollection (buildings + roads)
├── simulation.json     # Simulation state snapshot
└── metadata.json       # Bbox, city name, cache date, OSM data attribution
```

### `manifest.json` schema
```json
{
  "format_version": 1,
  "app_version": "1.0.0",
  "created_at": "2025-06-01T12:00:00Z"
}
```

### `simulation.json` schema
```json
{
  "clock_time": 3600,
  "speed": 1.0,
  "agent_count": 100,
  "agents": [
    { "id": "car_001", "position": [x, y, z], "path": ["edge_id", ...], "state": "moving" }
  ],
  "active_events": [
    { "type": "road_closure", "entity_id": "way_123456", "triggered_at": 1200 },
    { "type": "power_outage", "entity_id": "way_789012", "triggered_at": 900 }
  ]
}
```

### Versioning Policy
The `format_version` field enables forward compatibility. If a newer app version opens an older `.city` file, it runs a migration transform. If an older app opens a newer format version, it displays a "please update the app" error rather than silently loading corrupt state.
